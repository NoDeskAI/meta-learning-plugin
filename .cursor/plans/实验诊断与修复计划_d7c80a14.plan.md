# 实验诊断与修复计划

## 诊断结论

### 1. 网络问题：存在，但不是主因

终端日志中发现以下网络问题：

- **4 次 Nodesk 500 错误**：`HTTP/1.1 500 Internal Server Error`（行 25, 90, 117, 244）
- **3 次 Nodesk 超时**：`Request timed out.`（working_directory treatment trial 1 的 agent 调用和 eval 均超时）
- **2 次 DashScope 断连**：`materialize_signal LLM call failed`（行 70）和 `evaluate_skill_update LLM call failed: Server disconnected`（行 340）

**影响范围**：
- working_directory treatment trial 1：agent 仅执行了 2 次迭代就因超时中断（只跑了 `pwd`），LLM eval 也超时返回 null
- 其他场景中 500 错误导致个别迭代失败后重试成功
- DashScope 断连触发了 fallback 逻辑，未影响最终 materialize 结果

**结论**：网络不稳定确实损害了 1 个 trial 的结果，但 **不是主要失败原因**。working_directory 和 git_workflow 的 treatment 失败是 taxonomy 内容问题（过于抽象，未包含足够具体的行为指令），不是网络问题。

### 2. 代码链路错误：发现 3 处 bug

**Bug 1：cross-scenario `specific_exceptions` 检查路径不完整** (`interactive_meta_test.py:926-934`)

```python
scraper_paths = [
    Path(ws) / "projects" / "web-scraper" / "scraper.py",
    Path(ws) / "scraper.py",
]
```

agent 实际创建文件在 `Path(ws) / "web-scraper" / "scraper.py"`，但该路径不在搜索列表中。

**Bug 2：cross-scenario `specific_exceptions` 正则不匹配常见 requests 异常** (`interactive_meta_test.py:937-944`)

```python
r"\bexcept\s+(requests\.\w+|ConnectionError|HTTPError|TimeoutError|..."
```

- `Timeout` (requests.exceptions.Timeout) 不匹配 `TimeoutError`
- `URLRequired` 不匹配 `URLError`
- `RequestException` 不在模式中
- 且 agent 代码中有 `except Exception as e:` 导致 `generic > 0`，直接判定失败

**Bug 3：cross-scenario `pip_verify` 不匹配 `pip3 show`** (`interactive_meta_test.py:949-952`)

```python
checks["pip_verify"] = any(
    "pip show" in cmd.lower() or "pip list" in cmd.lower()
    for cmd in exec_result["shell_commands"]
)
```

agent 执行的命令是 `pip3 show requests beautifulsoup4`，但 `"pip show"` 不是 `"pip3 show"` 的子串。需要同时匹配 `pip3 show` 和 `pip3 list`。

**实际影响**：cross-scenario 中 agent 其实写了较好的异常处理代码且执行了 `pip3 show`，但因为检查逻辑 bug 导致 2/4 规则被误判为不通过。

### 3. temperature=0 是否合理

**合理**。理由：
- 实验目标是测量 taxonomy 注入的因果效应，需要控制变量
- temp=0 确保同一 prompt 下 agent 行为尽量一致，排除随机性干扰
- 实际观察：3 次 baseline trials 行为高度一致（全部在 CWD 创建项目、全部不验证安装），说明 temp=0 有效控制了噪声
- 唯一问题是 minimax-m2.7 在 temp=0 下对 system prompt 中的抽象规则响应不敏感（working_directory/git_workflow），但这是 taxonomy 内容问题，不是 temperature 问题

### 4. meta-learning LLM 调用

**成功但质量参差**：
- 4 个场景均成功完成 Layer 2 全流程（materialize -> cluster -> taxonomy -> skill）
- 1 次 materialize LLM 调用失败触发 fallback（影响不大，因为 4 个信号中 3 个成功足以聚类）
- 2 次 `evaluate_skill_update` 失败（DashScope 断连），但因为 taxonomy 已在之前步骤生成，对实验结果无影响

**质量问题**：DashScope qwen3.5-plus 生成的 taxonomy 对于 working_directory 和 git_workflow 过于抽象（例如 "Scan the conversation history for explicit path constraints"），agent 无法据此执行具体行为。而 code_style 和 verification_discipline 的 taxonomy 较具体（"use specific exceptions like FileNotFoundError"），效果好。

### 5. 评估 LLM 调用

**基本成功，但有 2 个问题**：
- 1 次超时（working_directory trial 1，返回 `null`）
- code_style trial 1：auto-check 通过（`bare=0, specific=2`），但 LLM eval 判定不合规。这是因为 LLM 观察到代码中仍有 `except Exception` 子句（agent 同时使用了 specific + generic），而 auto-check 的 `_autocheck_code_style` 函数只检查 `bare == 0 and specific > 0`，不检查 `generic`。LLM 的判断更严格但更准确。

**建议**：`_autocheck_code_style` 应同时检查 `generic == 0`，使其与 `_check_code_style` 逻辑一致。

### 6. LLM 切换

将做以下修改：

- **评估 LLM**：`openai/gpt-4o` -> `anthropic/claude-opus-4.6`（通过 Nodesk gateway）
- **Meta-learning LLM**：DashScope `qwen3.5-plus` -> Nodesk gateway `minimax-m2.7`

涉及的修改点：

**文件 1：[scripts/interactive_meta_test.py](scripts/interactive_meta_test.py)**
- 行 58：`EVAL_MODEL = "openai/gpt-4o"` -> `EVAL_MODEL = "anthropic/claude-opus-4.6"`
- 行 100-102：meta-learning LLM 环境变量从 DashScope 切换到 Nodesk
  ```python
  os.environ.setdefault("META_LEARNING_LLM_API_KEY", os.environ["NODESK_API_KEY"])
  os.environ.setdefault("META_LEARNING_LLM_BASE_URL", NODESK_GATEWAY)
  ```
- 删除对 `DASHSCOPE_API_KEY` 的强制检查（行 97-99），因为不再需要

**文件 2：[abtest/config.interactive.yaml](abtest/config.interactive.yaml)**
- 行 41：`model: "qwen3.5-plus"` -> `model: "minimax-m2.7"`
- 行 82 `enable_thinking: False` 在 [llm_openai.py](src/meta_learning/shared/llm_openai.py) 中的 `_chat` 方法需要确认 minimax-m2.7 是否支持此参数（若不支持需移除或做条件判断）
