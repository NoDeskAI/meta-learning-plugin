# C-B-A: Meta-Learning 架构重定位、任务选择与诊断链修复

## 现状诊断（基于 drawio + 实验数据）

drawio 展示的完整链路（Layer1 信号捕获 -> Layer2 物化/聚类/Taxonomy -> Layer3 离线挖掘 -> 反馈回 QuickThink + Agent 执行）设计上是正确的。但在 airline:35 上的 warmstart 实验（A 20% vs B 10%，p=0.58）暴露了三个层次的问题：

1. **信号太粗糙**: 只有 `reward=0`，LLM 无法准确诊断"到底哪一步做错了"
2. **任务不匹配**: airline:35 的主要失败原因是"参数精度"（订错航班），而非"策略错误"（meta-learning 的能力范围）
3. **诊断链断裂**: session 上下文中工具调用结果被截断，materialize prompt 缺乏引导

---

## Phase C: 架构重定位 — Meta-Learning 能做什么、不能做什么

### C1. 重新定义 meta-learning 的能力边界

Meta-learning 是**策略与流程学习系统**，不是参数优化系统：

- **CAN learn**: 工具调用顺序、策略选择（拒绝/接受）、多步流程、政策合规
- **CANNOT learn**: 精确参数（具体航班号）、数值计算（金额）、数据库状态匹配

### C2. 信号模型重新定义

当前信号来源和诊断能力：

```
reward=0 (太粗糙) ─────── 只知道失败，不知道为什么
conversation (可用) ────── LLM 理论上可以自行诊断，但当前 prompt 不够强
tool_calls (可用) ──────── 包含 args + results，但被过度截断
db_check (不可用) ──────── 评估系统内部，生产环境拿不到
```

引入**中间信号层** — NL Assertion 评估结果作为"QA 监督反馈"：

- tau-bench 每个 airline 任务都有 NL Assertions（如 "Agent does not cancel the reservation"）
- 这些断言由 LLM 读取对话后判定 pass/fail，描述**可观测行为**
- 在当前评估设置 (`EvaluationType.ALL`) 下，airline 任务**不评估** NL assertions
- 切换到 `ALL_WITH_NL_ASSERTIONS` 后，NL assertions **不影响 reward 计算**（因为 airline 的 `reward_basis` 只有 DB+COMMUNICATE），但产出每条断言的 pass/fail

**架构定位**: NL assertion 失败结果 = 生产环境中"人类质检员审阅对话后的反馈" = `user_corrections` 信号。这在 [`signal_capture.py`](src/meta_learning/layer1/signal_capture.py) 中映射为 `USER_CORRECTION` 触发类型（优先级第2），目前在 A/B 实验中完全未被使用。

### C3. 在代码中固化边界

在 [`src/meta_learning/shared/models.py`](src/meta_learning/shared/models.py) 的 `TaskContext` 中添加注释/文档，明确哪些字段来自 agent 可观测数据，哪些来自评估系统。在 [`README.md`](README.md) 中增加架构定位小节。

## Phase B: 评估任务选择 — 挑选 meta-learning 能发挥作用的任务

### B1. 为什么 airline:35 不适合

Task 35 的 3 条 NL assertions:
- "Agent does not cancel the reservation" — **策略层**（meta-learning 可帮助）
- "Agent books flight HAT069+HAT258" — **参数层**（meta-learning 难以帮助）
- "Agent charges $290 to credit_card_907483" — **参数层**

agent 在实验中策略层（拒绝取消）通常做对了，失败原因是参数层（订错航班），这恰好是 meta-learning 的盲区。

### B2. 推荐任务组合

选择标准：NL assertions **策略层占比高**（而非参数层）、meta-learning 可改善的失败模式、任务多样性。

**NL assertions 策略层 vs 参数层分析**（关键发现）：
- Task 35 只有 33% 策略层（1/3）— 之前一直用这个
- Task 14 只有 20% 策略层（1/5）— 之前推荐的，已排除
- 最佳候选任务策略层占比 100%

| Task | NL | 策略层占比 | Acts | 核心考验 | meta-learning 可学习的模式 |
|------|-----|----------|------|----------|--------------------------|
| **2** | 4 | **100%** | 3 | 补偿验证纪律：不主动越权、先验证再行动、核实用户声明 | "verify before act" 模式 |
| **9** | 4 | **100%** | 1 | 多规则政策合规：不同取消原因对应不同规则 | "multi-rule discrimination" 模式 |
| **18** | 6 | **83%** | 5 | 批量操作完整性：5个预订全部降舱 | "batch completeness" 模式 |

三个任务覆盖了三种独立的、meta-learning 确实能帮助学习的策略失败模式：
1. **Task 2**: "不验证就不行动"（verification discipline）
2. **Task 9**: "不同情况适用不同规则"（rule discrimination）
3. **Task 18**: "批量操作不要遗漏"（batch completeness）

### B3. 实验设计

修改 [`scripts/ab/run_trend_ab_experiment.py`](scripts/ab/run_trend_ab_experiment.py)：

- 添加 `--tasks` 参数（逗号分隔的 task ID），替代硬编码的 `airline:35`
- 每个任务运行 `--runs` 次（默认 5）
- A 组在**所有任务**共享同一个 meta-learning workspace（跨任务积累经验）
- B 组不注入任何 meta-learning
- 每个任务之间触发 Layer2（确保新经验被整合）

命令示例：
```bash
python scripts/ab/run_trend_ab_experiment.py \
  --tasks 2,9,18 --runs 5 --layer2-every 5 --skip-claw \
  --run-id phase_CBA_multitask
```

## Phase A: 诊断链修复 — 让 meta-learning 拿到有用的失败信号

### A1. 启用 NL Assertion 评估（不改变 reward）

修改 [`DeskClaw-Arena/src/agents/run_nobot_real.py`](../DeskClaw-Arena/src/agents/run_nobot_real.py)：

- `pick_evaluation_type` 对 airline 也返回 `ALL_WITH_NL_ASSERTIONS`
- 在 result dict 中添加 `nl_assertions` 字段（每条断言的 text + met/not_met）

```python
def pick_evaluation_type(domain: str):
    from tau2.evaluator.evaluator import EvaluationType
    return EvaluationType.ALL_WITH_NL_ASSERTIONS
```

### A2. 将 NL assertion 失败注入 meta-learning 信号捕获

修改 `_extract_signal_info` in [`run_trend_ab_experiment.py`](scripts/ab/run_trend_ab_experiment.py)：

- 从 `task_row` 中提取 `nl_assertions` 结果
- 将 **failed** assertions 转为 `user_corrections` 列表
- 例如: NL assertion "Agent should check that the flight was indeed delayed" failed → `user_corrections=["QA feedback: Agent should check that the flight was indeed delayed — but did not"]`
- **只注入策略层断言的失败**（如 "should not offer" / "should check" / "does not cancel"），过滤掉参数层断言（含具体航班号、金额的）

这使 `SignalCapture._determine_trigger` 命中 `USER_CORRECTION`（优先级第2），产生更精确的信号。

### A3. 改善 session 上下文质量

修改 [`src/meta_learning/shared/io.py`](src/meta_learning/shared/io.py) 的 `read_session_context`：

- 工具调用条目不再截断到 200 字符，改为截断到 500（保留 tool call 的 args 和 results）
- 这确保 materialize LLM 能看到 agent 实际调用了什么工具、传了什么参数、得到什么结果

### A4. 改善 materialize prompt

修改 [`src/meta_learning/shared/llm_openai.py`](src/meta_learning/shared/llm_openai.py) 的 `materialize_signal`：

- 在 prompt 中明确引导 LLM: "比较 agent 的工具调用参数与用户的原始需求，找出偏差"
- 在 prompt 中包含 `user_feedback` 字段（来自 NL assertion 失败）作为诊断线索
- 改进 `task_type` 分类，增加 `"customer_service"` 类型

### A5. 多任务 taxonomy 泛化

当使用多个不同任务时，taxonomy 需要能泛化。修改 taxonomy extraction prompt 增加指导：
- 从具体任务中提炼通用的策略规则
- 例如: 从 task 7 和 task 18 中都看到"agent 遗漏步骤"，应提炼为"multi-step verification pattern"

## 实验执行与验证

1. **Smoke test**: 对 task 2 单独跑 2 轮 A+B，验证 NL assertions 数据正确流入 signal（task 2 的 4 条 NL 全是策略层，最适合验证）
2. **Full run**: 3 tasks x 5 runs x 2 groups = 30 runs（约 30 x 3.5min ≈ 105min）
3. **分析维度**:
   - 每任务 A vs B 成功率（reward=1.0 的比例）
   - NL assertion 通过率对比（A 组在后期是否通过更多 strategy-level assertions）
   - Taxonomy 质量（是否产出可操作的策略指导，而非笼统的建议）
   - 信号质量（trigger=USER_CORRECTION 的信号是否包含 NL assertion 失败信息）
